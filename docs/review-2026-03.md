This report is a codebase-level audit focused on architecture/cohesion, DRY quality, API contract alignment, security/privacy posture, error handling, testing resilience, and documentation correctness.

## Findings (Ordered by Severity)

## P1

No P1 findings identified in this pass.

## P2

### P2-1: `update` command orchestrator is oversized and multi-responsibility

- Evidence: `src/commands/update.ts` (~330 LOC) currently mixes target discovery, metadata read policy, selector resolution, retry-token caching, install execution, JSON/human rendering, and summary accounting.
- Impact: future behavior changes (for example private/public selection policy or retry rules) are high-risk because state handling and presentation concerns are tightly coupled.
- Recommended fix path:
  1. Extract `runUpdateBatch` service layer (`lib/update/workflow.ts`) for execution.
  2. Move rendering (`table` + summary + json envelope conversion) to formatter module.
  3. Keep `runUpdateCommand` as thin adapter (`flags -> config -> workflow -> output`).

### P2-2: `search` command still carries multiple concerns in one unit

- Evidence: `src/commands/search.ts` handles flags, auth gating, pagination continuation caching, row indexing logic, table rendering, command hints, and API call orchestration in a single module.
- Impact: table UX changes and pagination behavior changes can unintentionally regress each other because of intertwined flow.
- Recommended fix path:
  1. Extract pagination cache management to `lib/search/pagination.ts`.
  2. Extract human renderer to `lib/search/render.ts`.
  3. Keep command module focused on control flow and error mapping only.

### P2-3: Fallback-to-beta behavior depends on backend message text

- Evidence: `src/lib/use/workflow.ts` `shouldFallbackToBeta` checks regex `/channel not set/i` against error message text, even though status/code is also inspected.
- Impact: backend message wording changes can silently disable fallback behavior for default `use` installs.
- Recommended fix path:
  1. Introduce machine-stable backend error details marker for unset channel.
  2. Switch fallback check from message regex to detail key.

## P3

### P3-1: Command modules repeat similar output/error scaffolding

- Evidence: `src/commands/view.ts`, `src/commands/history.ts`, `src/commands/use.ts`, and `src/commands/search.ts` each define local `printJson` and similar catch blocks.
- Impact: low immediate risk, but incremental drift in user-facing error format becomes more likely.
- Recommended fix path:
  1. Introduce lightweight shared command presenter utilities (`printJson`, error formatter helpers).
  2. Keep command-specific context while centralizing repeated templates.

### P3-2: README command examples are broad but not lifecycle-oriented

- Evidence: `README.md` documents commands and examples well but does not provide a single “public -> private -> install -> update” end-to-end contributor flow.
- Impact: onboarding remains command-centric rather than workflow-centric.
- Recommended fix path:
  1. Add short “Contributor lifecycle walkthrough” section linking to deep docs.
  2. Keep README concise by linking heavy detail to docs.
